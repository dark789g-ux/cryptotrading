# 中文文本编辑乱码复盘与规范

## 背景

项目运行在 Windows 11 环境，终端默认存在 GBK 语境，但仓库源文件要求统一使用 UTF-8。  
这意味着：**终端显示乱码**、**文件实际编码错误**、**编辑工具按错误字节边界替换**，是三种不同问题，必须分开判断。

本仓库已经出现过两类真实事故：

1. 编辑含中文的文档时，局部替换切断了 UTF-8 多字节字符，导致相邻汉字变成 `��`、`���` 等乱码。
2. 修复样式时重写了 `Sidebar.vue`，菜单文案被错误编码写回，页面实际展示成乱码文本，而不是单纯的终端显示异常。

## 根因总结

### 1. 终端乱码不等于文件乱码

- PowerShell / Windows 终端在 GBK 环境下读取 UTF-8 文件时，可能把正常中文显示成乱码。
- 这种情况下，浏览器页面里不一定有问题，文件本身也未必损坏。
- 因此不能只凭终端输出判断“文件坏了”，要结合页面表现和编辑前后差异一起判断。

### 2. 含中文文件做局部替换时，容易切坏 UTF-8 字节边界

- 中文字符在 UTF-8 中通常占多个字节。
- 如果替换工具的匹配边界刚好落在多字节字符中间，可能把相邻文本一起破坏。
- 同一文件连续多次小范围替换时，风险更高。

### 3. 中文密集文件不适合高频碎片化编辑

- 像 `AGENTS.md`、文档、菜单文案、表单文案这类文件，中文密度高，局部替换的容错率低。
- 一旦终端本身显示乱码，继续按“看到的乱码文本”去做二次替换，容易把问题扩大。

## 项目规范

### 一、先区分“显示乱码”还是“文件乱码”

遇到中文异常时，按以下顺序判断：

1. **先看页面/实际渲染结果**
   - 浏览器里正常、终端里异常：优先判断为终端显示问题。
   - 浏览器里也异常：优先判断为文件内容已损坏。

2. **再看修改范围**
   - 如果异常刚好出现在最近改过的中文字符串附近，优先怀疑编辑过程切坏了字符边界。
   - 如果异常普遍存在于终端输出，但页面无问题，不要贸然批量重写文件。

3. **不要把终端里看到的乱码文本当成真值继续编辑**
   - 终端显示的乱码经常不是文件中的真实字节序列。
   - 直接拿这段乱码去做下一轮替换，容易造成二次损坏。

### 二、编辑含中文文件的操作规范

1. **中文密集文件优先整段重写，不要频繁逐词替换**
   - 适用场景：文档、说明文件、菜单文案、页面标题、长段落提示语。
   - 原则：宁可整段替换，也不要围绕几个汉字做小范围 patch。

2. **局部替换时，匹配范围必须覆盖完整语义块**
   - `old_string` 要尽量覆盖整行、整段或完整对象项。
   - 不要只截取一小段中文词组，更不要卡在汉字中间。

3. **结构稳定时优先按结构替换，不按乱码文本替换**
   - 例如优先替换整个 `menuOptions` 数组项，或整个段落，而不是替换终端里显示出的乱码片段。

4. **修改中文文案后，必须立刻回读关键位置**
   - 至少检查：
     - 修改的字符串本身是否正常
     - 相邻未修改中文是否被连带破坏
     - 页面实际展示是否正常

5. **如果发现同一文件已有明显乱码，优先整文件重写**
   - 不要在受损文本上继续做局部修修补补。
   - 先恢复成一份明确的 UTF-8 正常内容，再继续后续修改。

### 三、哪些场景要提高警惕

以下场景默认视为高风险：

- `.vue` 文件中的中文 `template` 文案
- TypeScript 对象里的中文 `label` / `title` / `message`
- Markdown / 规范文档
- 含大量中文注释的文件
- 刚在 Windows 终端里已经显示过乱码的文件

## 本次侧边栏事故的经验

以 `apps/web/src/components/layout/Sidebar.vue` 为例：

- 页面中侧边栏菜单实际显示乱码，说明问题不是“终端看错”，而是文件内容已经被写坏。
- 菜单文案集中在 `menuOptions` 数组中，属于**结构集中、文本短、中文密集**的典型高风险区域。
- 正确做法不是继续按终端里的乱码字符串逐条替换，而是**整段重写 `menuOptions`，必要时整文件重写**。

## 执行清单

以后只要编辑中文文本，默认执行下面这份清单：

- 修改前先判断是终端显示问题还是文件真实损坏。
- 中文密集文件优先整段/整文件重写。
- 局部替换时保证匹配完整语义块，不截断汉字。
- 修改后立即回读相邻行。
- 涉及页面文案时，优先以浏览器实际显示为最终验收标准。

## 结论

本项目里的中文乱码问题，本质上不是“中文不能改”，而是：

- UTF-8 文件在 GBK 终端里容易误判；
- 中文多字节字符不适合碎片化替换；
- 一旦出现异常，必须用“结构化重写 + 立即回读”的方式处理。

后续凡是编辑中文文案、文档、菜单、标题，统一按本规范执行。
