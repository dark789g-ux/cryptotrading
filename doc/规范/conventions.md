# 关键约束与编码规范

本文档列出 `cryptotrading` 项目的强制性约束，所有代码与文档修改都应遵守。

---

## 约束一览

| 约束域 | 规则 |
|--------|------|
| **文件编码** | 所有源代码与文档文件统一使用 UTF-8；涉及文件 I/O 时显式指定编码。 |
| **Windows 终端** | 在 GBK 终端中看到中文异常时，先区分“显示乱码”与“文件乱码”，不要直接按终端乱码继续替换。 |
| **中文文本编辑** | 中文密集文件优先整段或整文件重写，避免碎片化局部替换切坏 UTF-8 多字节字符。 |
| **编辑后检查** | 任何中文文案修改后，必须回读关键位置，确认修改文本与相邻中文都未损坏。 |
| **UI 文案修复** | 如果页面实际展示已经乱码，优先按结构整段重写，不在受损文本上继续逐词修补。 |

---

## 中文文本编辑规范

### 1. 判断问题类型

遇到中文异常时，先做下面的区分：

- 终端里乱码，但浏览器页面正常：优先判断为终端显示问题。
- 浏览器页面也乱码：优先判断为文件内容已损坏。

不要把终端里显示的乱码文本当成后续替换的依据。

### 2. 编辑策略

- 文档、菜单文案、页面标题、提示语这类中文密集区域，优先整段重写。
- 仅在结构明确、范围完整时才做局部替换。
- 局部替换时，匹配范围必须覆盖完整语义块，不要卡在中文字符中间。

### 3. 修改后检查

每次修改中文后，至少确认以下三点：

1. 目标文本本身正常。
2. 相邻中文没有被连带破坏。
3. 如果是 UI 文案，页面实际显示正常。

### 4. 高风险文件

以下文件默认按高风险处理：

- `.vue` 模板里的中文文案
- TypeScript 配置对象中的 `label`、`title`、`message`
- Markdown 文档
- 含大量中文注释的文件

### 5. 参考案例

已知案例：

- `doc/规范/edit-tool-cjk-mojibake.md`：中文文档编辑时切坏 UTF-8 多字节字符。
- `apps/web/src/components/layout/Sidebar.vue`：侧边栏菜单文案被错误编码写回，导致页面实际显示乱码。

处理这类问题时，统一遵循“结构化重写、修改后回读、以页面表现验收”。
