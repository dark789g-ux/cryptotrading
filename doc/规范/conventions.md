# 关键约束与编码规范

本文档列出 `cryptotrading` 项目的强制性约束，所有代码与文档修改都应遵守。

---

## 约束一览

| 约束域 | 规则 |
|--------|------|
| **文件编码** | 所有源代码与文档文件统一使用 UTF-8；涉及文件 I/O 时显式指定 `encoding='utf-8'`。 |
| **Windows 终端** | 在 GBK 终端中看到中文异常时，先区分“显示乱码”与“文件乱码”，不要直接按终端乱码继续替换。 |
| **中文文本编辑** | 中文密集文件优先整段或整文件重写，避免碎片化局部替换切坏 UTF-8 多字节字符。 |
| **编辑后检查** | 任何中文文案修改后，必须回读关键位置，确认修改文本与相邻中文都未损坏。 |
| **UI 文案修复** | 如果页面实际展示已经乱码，优先按结构整段重写，不在受损文本上继续逐词修补。 |

---

## 中文乱码问题背景

项目运行在 Windows 11 环境，终端默认存在 GBK 语境，但仓库源文件要求统一使用 UTF-8。

因此要把下面三类问题分开判断：

1. 终端显示乱码。
2. 文件实际编码错误。
3. 编辑工具按错误字节边界替换，导致文件内容被写坏。

本仓库已经出现过两类真实事故：

1. 编辑含中文的文档时，局部替换切断 UTF-8 多字节字符，导致相邻汉字变成 `��`、`���` 等乱码。
2. 修复样式时重写 `Sidebar.vue`，菜单文案被错误编码写回，页面实际展示成乱码文本，而不是单纯的终端显示异常。

---

## 根因总结

### 1. 终端乱码不等于文件乱码

- PowerShell / Windows 终端在 GBK 环境下读取 UTF-8 文件时，可能把正常中文显示成乱码。
- 浏览器页面正常、终端异常时，优先判断为终端显示问题。
- 不能只凭终端输出判断“文件坏了”，要结合页面表现和编辑前后差异一起判断。

### 2. 局部替换可能切坏 UTF-8 字节边界

- 中文字符在 UTF-8 中通常占多个字节。
- 如果替换工具的匹配边界刚好落在多字节字符中间，可能把相邻文本一起破坏。
- 同一文件连续多次小范围替换时，风险更高。

### 3. 中文密集文件不适合碎片化编辑

- `AGENTS.md`、文档、菜单文案、表单文案这类文件中文密度高，局部替换的容错率低。
- 一旦终端本身显示乱码，继续按“看到的乱码文本”做二次替换，容易把问题扩大。

---

## 中文文本编辑规范

### 1. 先判断问题类型

遇到中文异常时，按以下顺序判断：

1. 先看页面或实际渲染结果：
   - 浏览器里正常、终端里异常：优先判断为终端显示问题。
   - 浏览器里也异常：优先判断为文件内容已损坏。
2. 再看修改范围：
   - 如果异常刚好出现在最近改过的中文字符串附近，优先怀疑编辑过程切坏了字符边界。
   - 如果异常普遍存在于终端输出，但页面无问题，不要贸然批量重写文件。
3. 不要把终端里看到的乱码文本当成真值继续编辑：
   - 终端显示的乱码经常不是文件中的真实字节序列。
   - 直接拿这段乱码去做下一轮替换，容易造成二次损坏。

### 2. 编辑含中文文件

1. 中文密集文件优先整段重写，不要频繁逐词替换。
   - 适用场景：文档、说明文件、菜单文案、页面标题、长段落提示语。
   - 原则：宁可整段替换，也不要围绕几个汉字做小范围 patch。
2. 局部替换时，匹配范围必须覆盖完整语义块。
   - `old_string` 要尽量覆盖整行、整段或完整对象项。
   - 不要只截取一小段中文词组，更不要卡在汉字中间。
3. 结构稳定时优先按结构替换，不按乱码文本替换。
   - 例如优先替换整个 `menuOptions` 数组项，或整个段落，而不是替换终端里显示出的乱码片段。
4. 如果发现同一文件已有明显乱码，优先整文件重写。
   - 不要在受损文本上继续做局部修补。
   - 先恢复成一份明确的 UTF-8 正常内容，再继续后续修改。

### 3. 修改后检查

每次修改中文后，至少确认以下三点：

1. 目标文本本身正常。
2. 相邻未修改中文没有被连带破坏。
3. 如果是 UI 文案，页面实际显示正常。

### 4. 高风险文件

以下文件默认按高风险处理：

- `.vue` 模板里的中文文案。
- TypeScript 配置对象中的中文 `label` / `title` / `message`。
- Markdown 文档。
- 含大量中文注释的文件。
- 刚在 Windows 终端里已经显示过乱码的文件。

## 事故经验

### 1. 中文文档编辑事故

含中文的规范文档曾在局部替换时切坏 UTF-8 多字节字符。处理这类文件时，不要围绕几个中文词做碎片化替换，应整段重写并立即回读。

### 2. 侧边栏文案事故

以 `apps/web/src/components/layout/Sidebar.vue` 为例：

- 页面中侧边栏菜单实际显示乱码，说明问题不是“终端看错”，而是文件内容已经被写坏。
- 菜单文案集中在 `menuOptions` 数组中，属于结构集中、文本短、中文密集的典型高风险区域。
- 正确做法不是继续按终端里的乱码字符串逐条替换，而是整段重写 `menuOptions`，必要时整文件重写。

---

## 执行清单

以后只要编辑中文文本，默认执行下面这份清单：

- 修改前先判断是终端显示问题还是文件真实损坏。
- 中文密集文件优先整段或整文件重写。
- 局部替换时保证匹配完整语义块，不截断汉字。
- 修改后立即回读关键位置和相邻行。
- 涉及页面文案时，优先以浏览器实际显示为最终验收标准。

---

## 结论

本项目里的中文乱码问题，本质上不是“中文不能改”，而是：

- UTF-8 文件在 GBK 终端里容易误判。
- 中文多字节字符不适合碎片化替换。
- 一旦出现异常，必须用“结构化重写 + 立即回读”的方式处理。

后续凡是编辑中文文案、文档、菜单、标题，统一按本规范执行。
