# Spec 文档助手

帮用户为单个模块创建或优化面向 AI 编程助手的 SPEC 文档，保存至项目 `doc/specs/` 目录。

## 判断模式

先判断用户意图：

- **新建**：用户要为尚未动手的功能写 SPEC
- **优化**：用户提供了已有 SPEC 文件路径或内容，要审查改进

如果不明确，用 `AskUserQuestion` 问一句："你是要新建一份 SPEC，还是优化已有的？"

---

## 新建模式

### 第一步：收集信息

用 `AskUserQuestion` 逐步追问，每次不超过 3 个问题，直到以下信息都清楚：

**必须搞清楚的：**
- 模块名称和一句话描述（做什么）
- 涉及哪些页面 / 组件 / 后端接口
- 涉及哪些数据库表（已有的还是新建的）
- 核心用户操作流程是什么
- 明确不做哪些事（防止 AI 过度发挥）
- 验收时怎么判断做好了

**可选但有价值的：**
- 参考哪些现有模块或文件（带路径）
- 有无特殊约束（权限、性能、第三方服务等）

追问策略：先问最关键的，根据回答决定后续问什么。如果用户回答里已经包含了某些信息，不要重复问。

### 第一步补充：发散确认潜在需求

基础信息收集完毕后，不要急着生成 SPEC。先站在"有经验的产品经理"视角，围绕用户描述的核心功能发散思考，从以下角度推演用户**可能需要但没主动提及**的需求：

- **关联操作**：有新增，是否需要编辑、删除、查看详情？有列表，是否需要搜索、筛选、排序？
- **状态与反馈**：加载中、空状态、操作成功 / 失败的提示
- **边界场景**：数据量大时的分页；极端值（空字符串、超长文本、零、负数）
- **权限 / 角色**：不同角色是否看到不同数据？某些操作是否需要鉴权？
- **数据一致性**：删除时有关联数据怎么处理？并发操作是否互斥？
- **可观测性**：是否需要操作日志、记录操作人和时间？
- **性能考量**：数据规模预期？是否需要索引、缓存、异步任务？
- **扩展点**：用户将来可能想要但现在不做的功能（有助于技术方案提前预留）

根据本次模块的实际情况，挑选其中真正有可能被遗漏的 5~10 条，用 `AskUserQuestion` 一次性呈现：

> 根据你描述的功能，我联想到以下可能需要的点，请确认哪些要纳入本次 SPEC（不确定的可以先标"待定"，不需要的直接说不做）：
>
> 1. …
> 2. …
> …

**注意**：只列真正有讨论价值的；显然不需要或用户已经明确说"不做"的不要列。用户确认后，将结果合并进后续的信息基础里。

### 第二步：生成 SPEC

信息收集完毕后，生成 SPEC 文档，保存到 `doc/specs/<模块名>.md`。

**文档结构：**

```markdown
# [模块名] SPEC

> 面向 AI 编程助手。最后更新：[日期]

## 1. 概述

[一句话说清楚这个模块做什么、为什么需要它]

## 2. 背景与上下文

- **相关文件**：[列出路径，例如 `apps/web/src/views/XxxView.vue`]
- **数据库表**：[涉及的表名]
- **依赖服务**：[外部 API、第三方库等，没有则省略]

## 3. 功能需求

每条格式：**[功能点]**：[触发条件] → [预期结果]

- **xxx**：用户点击 xxx 按钮 → 系统执行 xxx，页面显示 xxx
- ...

## 4. 明确不做的事

- 不支持 xxx（原因：xxx）
- 不处理 xxx 场景

## 5. 技术方案

### 后端

| 方法 | 路由 | 入参 | 出参 | 说明 |
|------|------|------|------|------|
| GET  | /api/xxx | - | `{ data[] }` | ... |

核心逻辑说明（如有复杂业务逻辑）

### 前端

- **页面/组件**：[文件路径或新建位置]
- **交互逻辑**：[关键交互描述]
- **状态管理**：[如果需要]

### 数据库

[新增或修改的表结构，用 SQL 或列表描述]

## 6. 验收标准

- [ ] [可验证的具体条件1]
- [ ] [可验证的具体条件2]
- ...
```

### 第三步：输出

文件保存后，告诉用户文件路径，并简要说明文档里哪些地方还比较模糊（如果有），建议用户确认后再交给 AI 编程。

## 优化模式

### 第一步：读取已有 SPEC

读取用户提供的 SPEC 文件。

### 第二步：审查

按以下维度检查，记录问题：

| 维度 | 检查点 |
|------|--------|
| **完整性** | 6 个章节是否都有？是否有空白或"待定"？ |
| **明确性** | 有无模糊词（"适当"、"合理"、"尽量"）？AI 能否无歧义地执行？ |
| **可验证性** | 验收标准是否可以逐条判断通过/不通过？ |
| **范围控制** | "不做的事"是否明确？技术方案有无遗漏或冲突？ |
| **上下文** | 相关文件路径是否具体？表结构是否清晰？ |

### 第三步：追问缺失信息

对于审查中发现的问题，用 `AskUserQuestion` 向用户追问，每次不超过 3 个问题。

### 第四步：输出改进后的 SPEC

将改进后的完整 SPEC 写回原文件（或用户指定的新路径）。在回复中列出主要改动点，方便用户快速 diff。

---

## 写作原则

- 用中文写 SPEC，技术术语（表名、字段名、路由）保持英文
- 每句话要让 AI 编程助手能直接执行，避免需要再次猜测
- 宁可写得具体，不要写得宽泛
- 文件路径要用项目实际路径，不要写占位符
