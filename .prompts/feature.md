# 任务：实现用户需求（新功能 / 改动）

## 流程

### 1. 需求澄清（产出：澄清清单 + 潜在需求清单）

**阅读代码（先读后问）**

- 通过 Glob / Grep / Read 阅读涉及页面或模块的现有实现，理解当前状态后再追问。

**向用户追问（AI 主动）**

- 使用 `AskUserQuestion` 对模糊点逐项追问；按优先级排序。

**伪需求识别**

- 主动识别「伪需求」：与现有能力重复、与既定架构冲突、投入产出过低；说明理由并给出替代方案。

**潜在需求清单（必须编号；用户可追问）**

- 推演用户未明确说出的潜在需求，分三类列出；**每一类内的条目从 1 开始连续编号**（见下方「输出格式」）。
- 列出后须说明：用户对**任意一条**若不理解，可直接说出类名 + 序号（例如「连带体验-2」），你必须用非术语化的简短说明解释该条在做什么、不选会怎样；用户追问未解决前，不得默认用户已理解并勾选。

**三类标签（仅作分类，编号在类内）**

- **连带体验**：如加载态、空态、错误提示、权限校验。
- **边界场景**：如极值、并发、离线、数据不存在。
- **后续衍生**：如可能要加的筛选 / 导出 / 批量操作。

**门禁**

- ⛔ 在用户对「澄清清单」与「潜在需求清单」给出明确回应（含逐项确认、或声明哪些采纳 / 不采纳、或完成对某几条追问后的结论）之前，不得进入方案设计。

---

### 2. 方案设计（产出：改动方案文档）

在用户提供的信息与澄清结论范围内交付：

- **改动文件清单**：文件路径 + 改动类型（新增 / 修改 / 删除）+ 核心职责。
- **核心逻辑**：接口契约（入参 / 出参 / 错误码）、数据流、关键算法伪代码。
- **边界处理**：对齐澄清阶段已确认的边界场景，逐条给出处理策略。
- **影响面**：会影响的现有功能、建议回归路径。
- **不做什么**：显式列出本次不处理项，避免范围蔓延。

**门禁**

- ⛔ 用户未确认方案前，不得动手写业务代码。

---

### 3. 执行（产出：可运行的变更）

**节奏（强制）**

- 按方案中的文件清单顺序连续编写；全部完成后统一汇报；禁止每完成一个文件就停下等待确认。

**偏差（强制）**

- 若实现中发现必须改动方案外的文件或逻辑：立即停笔；说明 ① 为什么必须改 ② 影响哪些现有功能；待用户确认后再继续。
- 禁止顺手修复无关 bug 或重构周边代码。

**完成交付（强制）**

- ⛔ 禁止由 AI 自行执行、运行、调试或验证功能（含 curl、启动服务、打开页面等）；一律由用户手动验证。

全部文件完成后，**按顺序**输出：

**1. 变更摘要**

每行一文件，格式：`[类型] 文件路径 — 做了什么`（类型：新增 / 修改 / 删除）

**2. 手动验证清单**

格式：`步骤 N：[操作] → [预期结果]`

必须覆盖：主流程 × 至少 2 个边界场景（空数据、异常输入、权限不足等）

**3. 数据库变更提示**（仅当有 Entity / Migration / SQL 改动时）

- 说明：改了哪张表、什么字段
- 命令示例：`docker exec crypto-postgres psql -U cryptouser -d cryptodb -c "..."`

**4. 服务重启提示**（仅当有后端文件改动时）

- 命令：`pnpm run dev` 或对应生产命令

---

## 澄清阶段「潜在需求」输出格式（强制）

分三块输出，**每块内条目带序号**：

```
## 潜在需求 · 连带体验
1. …
2. …

## 潜在需求 · 边界场景
1. …
2. …

## 潜在需求 · 后续衍生
1. …
2. …
```

用户若提问「某一条是什么意思」，回复时注明对应「类名 + 序号」，避免串项。

---

## 硬约束

- 需求澄清阶段：可只读业务代码（理解现状、核查伪需求），禁止写操作。
- 未完成「需求澄清」阶段，禁止写入任何业务代码。
- 每阶段产出须交付用户确认；用户未回复「继续 / 确认」或未完成对已列出条目的必要追问与结论时，不得进入下一阶段。
- 执行中若发现改动超出原需求（新增文件、跨模块重构、修改公共组件等），立即暂停并二次确认。

---

## 输入（将下面占位符替换为实际内容后使用；本节置于提示词末尾以便复制）

- **页面 / 模块**：<涉及的前端页面或后端模块，如 BacktestView、strategies.service>
- **需求描述**：<一句话：谁、在什么场景下、要做什么、达成什么结果>
- **当前状态**（可选）：<已有实现、相关 PR、已排除的方案>
- **验收标准**（可选）：<怎样算「做完了」>
