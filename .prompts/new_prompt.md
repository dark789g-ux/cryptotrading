# 任务：实现用户需求（新功能 / 改动）

## 流程

### 1. 需求澄清（产出：澄清清单 + 潜在需求清单）
- **先读代码**：通过 Glob / Grep / Read 阅读涉及页面或模块的现有实现，理解当前状态，再开始追问
- 用 `AskUserQuestion` 逐项追问模糊点；一次最多问 3 个，按优先级排序
- 主动识别**伪需求**（与现有能力重复、与既定架构冲突、ROI 过低），说明理由并给出替代方案
- 推演用户**未说出的潜在需求**，分三类列出供用户勾选：
  - 连带体验：如加载态、空态、错误提示、权限校验
  - 边界场景：如极值、并发、离线、数据不存在
  - 后续衍生：如可能要加的筛选/导出/批量操作
- ⛔ 用户未对清单逐项回复前，不得进入方案设计

### 2. 方案设计（产出：改动方案文档）
- **改动文件清单**：文件路径 + 改动类型（新增/修改/删除）+ 核心职责
- **核心逻辑**：接口契约（入参/出参/错误码）、数据流、关键算法伪代码
- **边界处理**：对应澄清阶段的边界场景逐条给出处理策略
- **影响面**：会影响的现有功能、需要回归测试的路径
- **不做什么**：显式列出本次不处理的相关项，避免范围蔓延
- ⛔ 用户未确认方案前，不得动手编码

### 3. 执行（产出：可运行的变更）

**节奏控制（强制）**
- 按方案文件清单顺序连续编写，全部完成后统一汇报；禁止每完成一个文件就停下等待确认
- 执行中保持静默，完成后在"变更摘要"中列出每个文件的改动说明

**偏差处理（强制）**
- 实现中若发现必须改动方案外的文件或逻辑 → 立即停笔 → 说明：① 为什么必须改 ② 影响哪些现有功能 → 等用户确认后再继续
- 禁止"顺手"修复不相关 bug 或重构周边代码

**完成交付（强制）**

⛔ **禁止行为**：AI 不得自行执行、运行、调试或以任何方式验证功能——包括 curl 请求、启动服务、打开页面等——一律交由用户手动验证。

全部文件完成后，**按顺序**输出以下内容：

**1. 变更摘要**
每行一文件，格式：`[类型] 文件路径 — 做了什么`（类型：新增 / 修改 / 删除）

**2. 手动验证清单**
格式：`步骤N：[操作] → [预期结果]`
必须覆盖：主流程 × 至少 2 个边界场景（空数据、异常输入、权限不足等）

**3. 数据库变更提示**（仅当有 Entity / Migration / SQL 改动时输出）
- 说明：改了哪张表、什么字段
- 命令：`docker exec crypto-postgres psql -U cryptouser -d cryptodb -c "..."`

**4. 服务重启提示**（仅当有后端文件改动时输出）
- 命令：`pnpm run dev` 或对应生产命令

## 硬约束
- 需求澄清阶段：可以**只读**业务代码（用于理解现状、核查伪需求），禁止任何写操作
- 未完成"需求澄清"阶段，禁止写入任何业务代码
- 每阶段产出必须显式交付给用户确认，用户未回复"继续/确认"不得进入下一阶段
- 执行中若发现改动超出原需求（新增文件、跨模块重构、修改公共组件），立即暂停并二次确认
