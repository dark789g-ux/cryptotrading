# 审查者 Prompt 模板（spec 合规 + 代码质量）

派发审查者子代理时使用本模板。**一个审查者一次性覆盖两个维度**：

1. **spec 合规** —— implementer 构建的正是被要求的东西（不多也不少）
2. **代码质量** —— 实现做得扎实（干净、有测试、可维护）

**spec 缺口优先于质量问题**：若 spec 不合规，先修 spec；spec 合规后再看质量。两者都须在本次审查中给出结论。

```
Task tool (general-purpose)：
  description: "Review Task N: [任务名]"
  prompt: |
    你正在审查一份实现。你要同时核实两件事：它构建的是否正是被要求的东西（spec 合规），
    以及它是否构建得扎实（代码质量）。

    ## 被要求的内容

    [任务需求全文 —— 直接粘贴，不要让审查者去读计划文件]

    ## implementer 声称构建了什么

    [来自 implementer 的汇报]

    ## 审查范围

    - 改动的文件：[本任务改动的文件清单]
    - BASE_SHA：[任务开始前的 commit]
    - HEAD_SHA：[当前 commit]
    （并行批次中，请按"改动的文件清单"界定审查范围，而非整段 git 区间——
      同批次其他任务的 commit 可能交错在区间内。）

    ## 关键：不要相信汇报

    implementer 完成得快得可疑。它的汇报可能不完整、不准确或过于乐观。
    你必须独立核实一切——读它真正写的代码，逐行把实际实现与需求对照。

    **不要** 轻信它说自己实现了什么、相信它关于完整性的声明、接受它对需求的解读。
    **要** 读真实代码、查它声称已实现却缺失的部分、找它未提及的多余功能。

    ## 第一部分：spec 合规

    **缺失的需求：**
    - 它是否实现了所有被要求的内容？
    - 是否有它跳过或漏掉的需求？
    - 它是否声称某处可用，实际却没真正实现？

    **多余/不必要的工作：**
    - 它是否构建了未被要求的东西？是否过度工程化、加了 spec 之外的"锦上添花"？

    **理解偏差：**
    - 它对需求的解读是否与本意不同？是否解决了错误的问题？
    - 是否实现了正确的功能但方式不对？

    **靠读代码来核实，而不是靠相信汇报。**

    ## 第二部分：代码质量

    在确认（或记录）spec 合规之后，评估实现质量：
    - 命名是否清晰准确（贴合"它做什么"，而非"它怎么做"）？
    - 代码是否干净、可维护？测试是否真正验证行为（而非只验证 mock）？
    - 每个文件是否有一个清晰的职责和定义良好的接口？
    - 各单元是否拆分得当，能被独立理解和测试？是否遵循计划中的文件结构？
    - 本次实现是否创建了一开始就很大的新文件，或显著膨胀了现有文件？
      （不要对既有的文件大小开炮——只关注本次改动贡献了什么。）

    ## 汇报格式

    分两段给出结论：

    **spec 合规：**
    - ✅ 符合 spec（代码检查后一切相符），或
    - ❌ 发现问题：[具体列出缺失或多余之处，附 file:line 引用]

    **代码质量：**
    - 优点
    - 问题（Critical / Important / Minor，各附 file:line）
    - 评估

    **总评：**
    - ✅ 通过（spec 合规 且 无未决的 Critical/Important 质量问题），或
    - ❌ 需修复：[按 spec 缺口优先、其次质量问题的顺序列出待办]
```
