## 目标

分析超长文件的内部结构，提出合理的模块化拆分方案，经用户确认后落地执行。

## 何时触发

**两种情形，都必须激活：**

1. **用户主动请求**：提到文件太长、需要拆分、模块化重构等
2. **编码完成后主动检测**：每次修改或新建文件后，统计目标文件行数。若任意文件超过 500 行，主动告知用户，例如：「`klineChartOptions.ts` 当前 680 行，超出 500 行限制，需要拆分吗？」

## 执行流程

### 第一步：读取并分析文件

读取目标文件，梳理清楚：

- 文件类型（Vue SFC / NestJS Controller·Service / TS 工具库 / 其他）
- 内部的自然边界：函数组、类、业务逻辑域、子组件等
- 各部分的职责和相互依赖
- 哪些内容可以独立抽出而不破坏对外接口

### 第二步：制定拆分方案并展示给用户

用清晰的树形结构或列表展示方案，让用户一眼看懂：

```
拆分方案：apps/web/src/composables/klineChartOptions.ts（当前 680 行）

拆分后：
├── klineChartOptions.ts（保留，约 100 行）—— 入口，组合并统一导出
├── klineChartBaseOption.ts（新建，约 180 行）—— grid / axis / tooltip 基础配置
├── klineChartSeries.ts（新建，约 220 行）—— 各 series 构建函数
└── klineChartGraphics.ts（新建，约 180 行）—— overlay 图形元素

Import 变更：
- KlineChartModal.vue 无需改动（原入口文件对外 API 保持不变）
```

方案需体现：新文件的路径和职责、原文件保留的内容、需要变更的 import 关系。

### 第三步：等待用户确认

展示方案后，**停下来等用户回复**，不得擅自执行。

提示语示例：「方案如上，是否按此拆分？或者需要调整文件划分？」

### 第四步：执行拆分

用户确认后，按方案操作：

1. 创建新文件，从原文件中提取对应内容
2. 修改原文件：删除已移走的部分，新增所需 import
3. 用 Grep 找出所有引用了被移动内容的文件，更新其 import 路径
4. 运行类型检查，确保无新增报错：
   - 前端：`cd apps/web && pnpm exec vue-tsc --noEmit`
   - 后端：`cd apps/server && pnpm exec tsc --noEmit`

### 第五步：汇报结果

简要说明：各新文件的路径和行数、原文件现在多少行、类型检查是否通过。

## 约束

- 只做结构性拆分，不额外重构逻辑、不新增功能
- 所有文件 UTF-8 编码
- 禁止引入 `any`，用 `unknown` + 类型收窄
- 若原文件有 `index.ts` barrel export，拆分后同步更新
- 拆分后每个文件单一职责、命名清晰反映其内容
