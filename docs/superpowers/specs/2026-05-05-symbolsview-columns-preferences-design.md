# SymbolsView 列配置设计

**日期**: 2026-05-05  
**状态**: 已确认，待实现  
**范围**: `apps/web` + `apps/server`

## 1. 目标

`SymbolsView.vue` 下面的两张表，用户可以自行选择显示哪些列，并调整这些列的显示顺序。配置需要跟随用户账号同步到服务端，而不是只保存在本地浏览器。

### 目标行为

- 用户在 `CryptoSymbolsPanel` 和 `ASharesPanel` 中分别管理列
- 每张表都支持“显示 / 隐藏”
- 每张表都支持“顺序调整”
- 配置按用户账号保存，换设备后仍然生效
- 新增列时，老用户能自动看到新列的默认状态，不会因为旧配置把新列永久藏掉

## 2. 设计原则

- 以当前用户为唯一归属，不做全局共享
- Crypto 和 A 股配置彼此独立
- 先做前端列配置，不把列裁剪下推到查询层
- 固定列保持固定，不允许隐藏或拖动
- 失败时回滚到上一次成功状态，并提示用户

## 3. 数据模型

### 3.1 服务端存储

新增一张用户偏好表，建议命名为 `user_preferences`。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | PK | 主键 |
| `user_id` | FK | 关联当前用户 |
| `key` | varchar | 偏好键，如 `symbols_view_columns` |
| `value` | jsonb | 偏好内容 |
| `updated_at` | timestamptz | 更新时间 |

约束：

- `user_id + key` 唯一
- `value` 只保存当前用户的 SymbolsView 列配置

### 3.2 配置结构

建议一个 key 下面保存两份配置，分别对应两个 tab：

```json
{
  "crypto": [
    { "key": "symbol", "visible": true },
    { "key": "close", "visible": true },
    { "key": "ma5", "visible": false }
  ],
  "aShares": [
    { "key": "tsCode", "visible": true },
    { "key": "name", "visible": true },
    { "key": "pe", "visible": false }
  ]
}
```

说明：

- `key` 表示列标识
- `visible` 表示是否显示
- 数组顺序就是用户看到的顺序
- 固定列仍会出现在配置里，但前端会强制保持显示且不可操作

## 4. 后端设计

### 4.1 新增模块

建议新增 `preferences` 或 `user-preferences` 模块，职责只做当前用户偏好读写。

### 4.2 API

#### `GET /api/preferences/symbols-view`

返回当前用户的 SymbolsView 列配置。

响应：

```json
{
  "crypto": [...],
  "aShares": [...]
}
```

#### `PUT /api/preferences/symbols-view`

保存当前用户的 SymbolsView 列配置。

请求体：

```json
{
  "crypto": [...],
  "aShares": [...]
}
```

### 4.3 服务端逻辑

- 从当前用户上下文读取 `userId`
- 读取时若无记录，返回默认配置
- 写入时做基础校验：
  - 数组必须存在
  - `key` 必须来自该页面的已知列集合
  - 固定列不可被标记为隐藏
  - 重复 `key` 只保留首次或直接判错，二选一实现时统一
- 返回前做一次规范化：
  - 去除未知列
  - 补齐默认列
  - 固定列强制 `visible: true`

## 5. 前端设计

### 5.1 组件划分

建议新增一套共享的列配置组件和配置适配层：

- `SymbolsView.vue`
  - 维持两个 tab
  - 负责切换 `crypto` / `aShares`
- `ColumnSettingsDrawer.vue`
  - 通用列设置抽屉
  - 支持勾选、上移、下移、恢复默认
- `useSymbolColumnPreferences.ts`
  - 负责加载、保存、规范化配置
  - 负责把“配置”映射成 `n-data-table` 的 `columns`
- `cryptoColumns.ts` / `aSharesColumns.ts`
  - 保存完整列元数据
  - 标记哪些列是固定列、默认显示列、可排序列

### 5.2 列元数据

每张表维护一份列定义注册表，至少包含：

- `key`
- `title`
- `width`
- `fixed`
- `sorter`
- `defaultVisible`
- `locked`：是否固定列，不允许隐藏和拖动

表格最终渲染时：

1. 先按配置顺序取列
2. 再补默认列中缺失项
3. 再追加固定列
4. 最后交给 `n-data-table`

### 5.3 交互

- 表格工具栏新增“列设置”按钮
- 点击后打开抽屉
- 抽屉左侧或上部展示可见列列表
- 每项支持：
  - 勾选显示 / 隐藏
  - 上移 / 下移
- 底部提供：
  - 恢复默认
  - 取消
  - 保存

### 5.4 保存策略

采用“本地立即生效，成功后再落库”的方式。

- 用户修改后，表格立刻重排
- 点击保存时请求服务端
- 服务端成功后同步当前状态
- 失败时回滚到上一次成功状态

## 6. 数据流

### 首次进入页面

1. `SymbolsView` 加载当前 tab
2. `useSymbolColumnPreferences` 请求用户配置
3. 如果服务端无配置，则合并默认配置
4. 渲染表格

### 用户修改列配置

1. 用户打开列设置抽屉
2. 勾选或调整顺序
3. 前端立即更新表格列
4. 点击保存
5. `PUT /api/preferences/symbols-view`
6. 成功后关闭抽屉
7. 失败后恢复旧配置并提示错误

### 用户切换 tab

- `crypto` 和 `aShares` 各自保留自己的配置
- 切换 tab 不会互相污染

## 7. 错误处理

- 配置加载失败：
  - 使用默认列配置兜底
  - 给出非阻塞提示
- 配置保存失败：
  - 回滚到上一次成功状态
  - 提示保存失败
- 服务端返回未知列：
  - 前端忽略
  - 不阻断页面展示
- 服务端返回缺失列：
  - 以默认配置补齐

## 8. 测试

### 后端

- 当前用户无配置时返回默认值
- 保存后可再次读回
- 未知列被忽略
- 固定列不能被隐藏
- 不同用户互不影响

### 前端

- 配置加载后能正确生成列顺序
- 隐藏列不渲染
- 固定列始终存在
- 保存失败能回滚
- Crypto / A 股 tab 各自独立

### 手工验证

- 登录用户 A，修改 Crypto 列配置
- 刷新页面后保持不变
- 换到用户 B，配置互不影响
- 切换 tab，两个表各自保持自己的列设置

## 9. 范围边界

不做的事情：

- 不把列配置下推到查询接口裁剪数据
- 不做跨页面共享列配置
- 不做服务端拖拽排序接口，先走整包保存
- 不改现有查询、筛选、排序语义

## 10. 验收标准

- 用户可在 `SymbolsView` 中配置列显示与顺序
- 配置跟随用户账号同步到服务端
- 重新登录或换设备后仍能恢复
- 两个 tab 独立保存
- 固定列保持可见
- 新增列不会把旧用户锁死在旧配置里
